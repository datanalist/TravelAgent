# Product Context — TravelAgent

## Зачем существует проект

Туроператоры теряют конверсию из-за медленной и неперсонализированной первичной коммуникации. В high-end сегменте ожидание клиента — мгновенный отклик, экспертный диалог, индивидуальный подход 24/7.

Существующие решения (CRM-чаты, виджеты) — только каналы доставки. Они не квалифицируют лиды, не ведут экспертный диалог, не адаптируют тон.

## Как система должна работать

### Основные сценарии

1. **«Хочу в тепло»** — неформальный запрос → уточнение бюджета/дат/стиля → `search_tours` → 2–3 варианта с объяснением
2. **«High-end клиент»** — formal register, premium-лексика, фокус на уникальности, без слова «скидки»
3. **«Работа с возражением»** — «дорого» → уточнение → альтернативы → логирование в CRM
4. **«Типовой вопрос (визы)»** → `get_policy_info` → структурированный ответ
5. **«Кросс-канал»** — начал в Telegram, продолжил в Web; контекст сохранён по `client_id`

### Граничные случаи

| # | Сценарий | Ожидаемое поведение |
|---|---|---|
| E1 | Пустое сообщение | Вежливая просьба уточнить |
| E2 | Вне travel-тематики | Small talk или редирект |
| E3 | Резкая смена темы | Сохранение контекста, уточнение |
| E5 | Противоречивые параметры (50к + 5* Бали) | Тактичное объяснение компромиссов |
| E6 | Prompt Injection | Валидация + system prompt hardening |
| E7 | Клиент «уходит» | Stage Tracker → follow-up |

## Пользовательский опыт

- Первый токен ответа — до 2 секунд (p50)
- Стриминг в Web (SSE), полный ответ в Telegram
- Диалог в одном окне: агент сам уточняет детали, предлагает туры, создаёт заявку
- Клиент не чувствует жёсткого скрипта — живой экспертный разговор

## Ключевые принципы продукта

- **No Hallucination**: агент никогда не называет цены/даты/отели без данных из `search_tours`
- **High-end тон**: без слова «скидки», фокус на качестве и уникальности
- **Данные — только из tools**: LLM reasoning, не оракул данных
- **Graceful degradation**: при ошибках — вежливый fallback, не сбой
