# TravelAgent — Продуктовое предложение (Product Proposal)

## 1. Обоснование идеи

### Прикладная задача

Туроператоры и турагентства получают поток трафика (сайт, Telegram, соцсети), но значительная часть конверсии теряется на этапе квалификации и первичной коммуникации. Менеджеры не успевают реагировать быстро, отвечают на типовые вопросы, не персонализируют предложения. В high-end сегменте ожидания к сервису особенно высоки: премиальный тон, скорость отклика 24/7, знание предпочтений клиента.

Существующие решения (чаты, виджеты, CRM) — это каналы доставки. Они:
- Не квалифицируют запросы автоматически
- Не собирают структурированный профиль клиента
- Не ведут экспертный диалог с персонализацией
- Не интегрируют подбор туров и CRM в единый поток
- Не адаптируют тон под сегмент (high-end vs mass)

### Почему агент, а не обычный чат-бот

Задача требует комбинирования нескольких типов рассуждений:
- **Интерпретация запросов** — открытые вопросы, неформализованные формулировки («хочу что-то теплое», «без пересадок»)
- **Квалификация лида** — определение бюджета, сроков, гибкости, вероятности покупки
- **Работа с возражениями** — «дорого», «не уверен», «посмотрю у конкурентов» требуют гибкого диалога
- **Подбор туров** — семантический поиск и ранжирование с объяснением выбора
- **Адаптация стиля** — high-end tone, формальность, выбор register (ты/вы)

Это классическая задача для LLM-агента с инструментами (search, CRM, policy) и памятью (session, profile).

### Почему сейчас

- Доступность LLM API (Claude, OpenAI, Mistral) с разумной стоимостью
- Зрелость tool-calling и streaming
- Рост digital-каналов (Telegram, Web) в travel-индустрии
- Давление на маржинальность — нужна автоматизация первичной коммуникации

## 2. Цель и метрики

### Цель проекта

Создать PoC мультиагентной системы AI-консьержа, которая квалифицирует лиды, ведёт экспертный диалог, подбирает туры и создаёт «подогретых» клиентов для менеджеров — с фокусом на high-end сегмент и продажную воронку.

### Продуктовые метрики

| Метрика | Целевое значение | Способ измерения |
|---|---|---|
| Конверсия из трафика в квалифицированный лид | >= 15% (vs ~5% baseline) | Сравнение до/после на тестовом трафике |
| Доля «подогретых» лидов (с заполненным профилем) | >= 70% от созданных лидов | Анализ записей в leads |
| Средний чек по лидам агента vs ручным | Не ниже (цель: рост за счёт апселла) | Сравнение при переходе в сделку |
| Снижение времени реакции (first response) | < 5 сек | Замер latency ответа |

### Агентные метрики

| Метрика | Целевое значение | Способ измерения |
|---|---|---|
| Точность Router (intent) | >= 85% | Ручная оценка на 100+ сообщениях |
| Корректность Decision Logic (стадия) | >= 80% | Анализ соответствия stage и контекста |
| Качество tool-calls | >= 80% вызовов корректны | Анализ логов |
| Соответствие high-end tone | >= 75% оценок «соответствует» | Ручная оценка ответов |

### Технические метрики

| Метрика | Целевое значение | Способ измерения |
|---|---|---|
| Latency p50 (ответ до первого токена) | <= 2 сек | Замер end-to-end |
| Latency p95 (полный ответ) | <= 15 сек | Замер end-to-end |
| Стоимость одного диалога (10 сообщений) | <= $0.15 | Подсчёт токенов |
| Uptime (в рамках демо) | >= 95% | Мониторинг |

## 3. Сценарии использования

### Основные сценарии

**Сценарий 1: «Хочу в тепло»**
Клиент пишет «хочу в тепло в феврале». Агент уточняет бюджет, город вылета, длительность, стиль (релакс/активный), затем вызывает `search_tours` и предлагает 2–3 варианта с объяснением.

**Сценарий 2: «High-end клиент»**
Клиент из сегмента high-end. Агент использует formal register, premium-лексику, фокус на уникальности и комфорте. Не предлагает «скидки», акцент на качестве и индивидуальности.

**Сценарий 3: «Работа с возражением»**
Клиент: «Дорого». Агент уточняет, что именно кажется дорогим, предлагает альтернативы (другое направление, даты, класс перелёта) и логирует взаимодействие в CRM.

**Сценарий 4: «Типовой вопрос (визы, документы)»**
Клиент спрашивает про визу в Турцию. Агент вызывает `get_policy_info`, даёт структурированный ответ, не переключая на менеджера.

**Сценарий 5: «Кросс-канал»**
Клиент начал в Telegram, позже зашёл в Web-чат. Сессия общая (по client_id), контекст сохраняется.

### Граничные случаи (Edge Cases)

| # | Сценарий | Ожидаемое поведение |
|---|---|---|
| E1 | Пустое/бессмысленное сообщение | Агент вежливо просит уточнить запрос |
| E2 | Запрос вне travel (политика, погода в городе и т.п.) | Small talk или вежливый редирект в тему путешествий |
| E3 | Резкая смена темы в середине квалификации | Агент сохраняет контекст, мягко уточняет приоритет |
| E4 | Множество требований одновременно | Агент структурирует, уточняет приоритеты, вызывает search_tours с учётом всех параметров |
| E5 | Противоречивые параметры (бюджет 50к, 5* отель на Бали) | Агент тактично объясняет реалистичность, предлагает компромиссы |
| E6 | Prompt Injection | Валидация ввода, system prompt hardening, агент не выполняет внешние инструкции |
| E7 | Клиент «уходит» — долго не отвечает | Stage Tracker переводит в follow-up, при новом сообщении — восстановление контекста |

## 4. Ограничения

### Технические ограничения

| Параметр | Значение |
|---|---|
| SLO: p95 latency | <= 15 секунд (включая LLM + tools) |
| Контекстное окно LLM | До 128K (реально используем до 16K на запрос с memory) |
| Session memory (Redis) | Summary + stage + scratchpad, TTL 24h |
| Max шагов агента на сообщение | 5 (tool-calls + финальный ответ) |
| Max токенов на ответ LLM | 2000 |
| Rate limit | 20 сообщений/минуту на client_id |

### Операционные ограничения

| Параметр | Значение |
|---|---|
| Бюджет на LLM API (весь PoC) | $100–200 |
| Целевая стоимость диалога (10 сообщений) | <= $0.15 |
| Команда | 2–3 человека |
| Срок разработки | 2–3 недели |
| Интеграция с CRM | Заглушка (таблица leads) |
| Поиск туров | Заглушка или локальная БД (500–1000 туров) |
| Каналы | Только Telegram + Web (один backend) |

## 5. Архитектурный эскиз

### Модули системы

```
┌─────────────────────────────────────────────────────────────────────┐
│                    Channels (Telegram + Web SSE)                      │
│         /telegram/webhook  |  /chat/stream                           │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────────┐
│                    API Gateway / FastAPI                             │
│              Auth, Rate limit, нормализация ChatMessage               │
└───────────────────────────────┬─────────────────────────────────────┘
                                │
┌───────────────────────────────▼─────────────────────────────────────┐
│              Orchestrator / Conversation Controller                   │
│    Резолв client/session, вызов Router + Decision + LLM              │
└──┬──────────────┬──────────────┬──────────────┬──────────────────────┘
   │              │              │              │
   ▼              ▼              ▼              ▼
┌──────┐  ┌──────────┐  ┌────────────┐  ┌─────────────────┐
│Router│  │ Decision │  │Memory Layer│  │ LLM Connector   │
│Intent│  │ Logic    │  │(Redis+PG)  │  │ (Claude/OpenAI) │
└──┬───┘  └────┬─────┘  └─────┬──────┘  └────────┬────────┘
   │           │              │                   │
   └───────────┴──────────────┴───────────────────┘
                                │
                    ┌───────────▼───────────┐
                    │      Tools Layer      │
                    │ search_tours          │
                    │ create_lead           │
                    │ get_client_profile    │
                    │ get_policy_info       │
                    └───────────────────────┘
```

### Описание модулей

| Модуль | Назначение | Тип |
|---|---|---|
| **Channels** | Telegram webhook, Web SSE — приём/отправка сообщений | Frontend/Adapter |
| **API Gateway** | Auth, rate limit, нормализация в internal request | Middleware |
| **Orchestrator** | Обработка сообщений, вызов Router → Decision → LLM | Оркестрация |
| **Router** | Intent classification (small talk, discovery, tour search, objections, policy) | LLM + эвристики |
| **Decision Logic** | Определение stage, action, tools_to_call | Rule-based или LLM |
| **Memory Layer** | Session (Redis: summary, stage), Profile (PostgreSQL) | Storage |
| **LLM Connector** | Единый интерфейс к Claude/OpenAI/Mistral, streaming, tool-calling | Integration |
| **Tools Layer** | search_tours, create_lead, get_profile, get_policy_info | Инструменты |
| **CRM Adapter** | Заглушка → таблица leads (MVP) | Integration |

### Интеграции

- **LLM API** (Claude / OpenAI / Mistral) — генерация ответов, tool-calling
- **PostgreSQL** — clients, sessions, messages, leads, itineraries
- **Redis** — session memory (summary, stage, scratchpad)
- **Telegram Bot API** — webhook для входящих сообщений

### Основной поток (MVP)

```
Клиент (Telegram/Web)
    │
    │  "Хочу в тепло в феврале, бюджет до 150к"
    ▼
┌─────────────────────────────────────┐
│  API: резолв client_id, session_id   │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  Router: intent = discovery          │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  Memory: загрузка summary, profile, │
│  stage                               │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  Decision: stage=discovery,          │
│  action=уточнить/подобрать,          │
│  tools=[search_tours]                │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  LLM: system prompt + memory + msg   │
│  → tool_call search_tours(...)       │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  Tools: search_tours → список туров  │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  LLM: финальный ответ + create_lead  │
│  (если квалифицирован)               │
└─────────────┬───────────────────────┘
              ▼
┌─────────────────────────────────────┐
│  Summarizer + Profile Updater        │
│  → Redis, PostgreSQL                 │
└─────────────┬───────────────────────┘
              ▼
Ответ стримится в Telegram/Web
```

### Разделение ответственности: LLM vs программная логика

#### Делегируется LLM / Агенту

| Задача | Обоснование |
|---|---|
| Интерпретация запроса и intent (частично) | Гибкое понимание естественного языка, синонимы, неформализованный ввод |
| Генерация ответа в нужном стиле | Адаптация тона, high-end лексика, персонализация |
| Работа с возражениями | Требует контекстуального рассуждения |
| Планирование tool-calls | Агент решает, когда вызывать search_tours, create_lead |
| Формирование объяснений выбора | «Почему предложил именно эти варианты» |

#### НЕ делегируется LLM (программная логика)

| Задача | Обоснование |
|---|---|
| Поиск туров (search_tours) | Детерминированный вызов БД/API, не допускает галлюцинаций по ценам и датам |
| Создание/обновление lead в CRM | Критично для целостности данных — только через структурированный tool |
| Проверка policy (визы, ограничения) | Детерминированная проверка по правилам компании |
| Резолв client_id/session_id | Идентификация по токену/chat_id |
| Rate limiting, auth | Системный уровень |
| Логирование и аудит | Системный уровень |
| Контроль количества шагов | Программный ограничитель |
