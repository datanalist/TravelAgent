# Спецификация Retriever

**Версия:** 1.0  
**Статус:** MVP  
**Связанные документы:** `docs/system-design.md`, ADR-002, `spec-memory-context.md`

---

## 1. Обзор

Retriever — слой получения данных для агента. В MVP используются два детерминированных источника:

| Источник | Tool | Тип данных |
|---|---|---|
| Каталог туров | `search_tours` | Туры, цены, наличие мест |
| База правил | `get_policy_info` | Визы, страховки, условия оператора |

Векторный/семантический поиск — **не MVP**, описан в разделе 8.

---

## 2. search_tours

### Параметры запроса (SearchParams)

| Параметр | Тип | Обязательный | Описание |
|---|---|---|---|
| `destination` | string | нет | Страна, регион, курорт |
| `departure_city` | string | нет | Город вылета |
| `date_from` | date | нет | Начало диапазона дат |
| `date_to` | date | нет | Конец диапазона дат |
| `duration_nights` | int | нет | Количество ночей |
| `budget_max` | int | нет | Максимальный бюджет, руб. |
| `hotel_stars` | int | нет | Категория отеля (1–5) |
| `travel_style` | enum | нет | `relax`, `adventure`, `family`, `bleisure` |
| `adults` | int | нет | Количество взрослых |
| `children` | int | нет | Количество детей |

### Формат ответа

```json
[{
  "tour_id": "string",
  "destination": "Мальдивы, Мале",
  "hotel": "Soneva Fushi 5*",
  "departure": "2026-02-10",
  "duration_nights": 10,
  "price_rub": 485000,
  "airline": "Emirates",
  "included": ["перелёт", "трансфер", "All Inclusive"],
  "available_seats": 4
}]
```

### Поведение при 0 результатах

Если `search_tours` возвращает пустой массив `[]`:
- Агент **не генерирует** альтернативные туры из «головы»
- Стандартный ответ: `«По вашим параметрам вариантов не найдено. Попробуйте изменить даты или бюджет»`
- Агент уточняет у клиента, какой параметр он готов изменить

### Актуальность данных

`search_tours` автоматически фильтрует туры с истёкшими датами вылета (R7). Устаревшие записи не попадают в выдачу.

## 3. get_policy_info

```python
get_policy_info(country: str, profile: ClientProfile) → PolicyInfo
```

### Что проверяет

- Визовые требования (страна назначения + гражданство клиента)
- Страховые требования (тип страховки, минимальное покрытие)
- Ограничения по авиакомпаниям
- Внутренние условия оператора: доплаты, условия отмены, переоформление

### Формат ответа (PolicyInfo)

```json
{
  "country": "Таиланд",
  "visa_required": false,
  "visa_notes": "Безвизовый въезд до 30 дней",
  "insurance_required": true,
  "insurance_min_coverage_eur": 30000,
  "airline_restrictions": [],
  "operator_conditions": {
    "cancellation_policy": "Возврат 100% при отмене за 30+ дней",
    "rebooking_fee_rub": 3000
  }
}
```

## 4. Ранжирование и усечение результатов

```
search_tours → [N результатов] → Orchestrator (top-N усечение) → LLM
```

- Если `search_tours` возвращает **> 10 результатов** → Orchestrator усекает до **top-N**
- Критерии ранжирования: релевантность параметрам запроса, цена (по умолчанию — возрастание)
- LLM видит **только усечённый список**
- LLM предлагает клиенту **2–3 варианта** из полученных результатов
- Конкретное значение top-N определяется при реализации Orchestrator (учитывая context budget)

---

## 5. Context Budget для Retrieval Results

| Ресурс | Лимит |
|---|---|
| Максимум токенов на запрос | 16 000 |
| Источник управления усечением | Orchestrator |

Orchestrator усекает результаты `search_tours` **до передачи LLM**, чтобы не превысить context budget. Передача сырого полного списка в LLM запрещена.

## 6. Принцип «No Hallucination» (ADR-002)

Агент **никогда** не называет цены, даты, отели и условия из «головы». Только данные из инструментов.

### Три уровня защиты

**Уровень 1 — System Prompt (декларативный)**
> «Не указывай цены и даты без данных из инструментов. Цитируй только результаты search_tours.»

**Уровень 2 — Программная проверка (детерминированный)**
- Если `tool_results` пустой → агент использует стандартный ответ «не найдено»
- Блок обработки пустого результата вынесен на уровень Orchestrator, не LLM

**Уровень 3 — Disclaimer в ответе (пользовательский)**
- Каждый ответ с ценами/датами содержит: `«Уточняйте актуальность и стоимость у менеджера»`

---

## 7. Ограничения и защита от злоупотреблений

| Ограничение | Значение |
|---|---|
| Размер локальной БД туров (MVP) | 500–1000 туров |
| Фильтрация устаревших туров | Автоматически в `search_tours` |
| Источник истины по ценам | Только `search_tours`, не LLM |
| Источник истины по правилам | Только `get_policy_info`, не LLM |

Агент не принимает данные о турах из сообщений пользователя как достоверные — только из инструментов.

---

## 8. Векторный поиск (будущее, не MVP)

**Не реализуется в MVP.** Планируется в следующих итерациях:

| Компонент | Назначение |
|---|---|
| pgvector / Qdrant | Семантический поиск по каталогу туров |
| Эмбеддинги клиентских профилей | Поиск похожих клиентов для рекомендаций |
| FAQ-поиск | Ответы на свободнотекстовые вопросы |

До реализации векторного поиска все запросы обрабатываются детерминированными инструментами (`search_tours`, `get_policy_info`).
